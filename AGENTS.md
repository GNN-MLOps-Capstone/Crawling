# AGENTS.md

이 문서는 `/home/dobi/Crawling`에서 작업하는 에이전트를 위한 저장소 전용 지침이다. 일반 원칙보다 이 문서의 지침을 우선한다.

## 1. 저장소 목적

- 이 저장소는 뉴스 수집/정제/적재/분석을 수행하는 `Airflow DAG`와 공용 파이프라인 모듈, 그리고 추천 결과를 제공하는 `FastAPI` API를 함께 포함한다.
- 핵심 디렉터리
  - `dags/`: Airflow DAG 정의와 SQL, 설정 파일
  - `dags/modules/`: DAG가 호출하는 공용 비즈니스 로직
  - `app/`: 추천 API 서버
  - `tests/`: 현재는 추천 API 중심 테스트
  - `docker-compose.yaml`, `dockerfile/`: 로컬 개발용 컨테이너 환경

## 2. 작업 우선순위

- 먼저 수정 대상이 `DAG 정의`인지, `공용 모듈`인지, `API 서버`인지 구분한다.
- 가능하면 DAG 내부에 로직을 직접 늘리지 말고 `dags/modules/`로 이동 또는 재사용한다.
- API 변경 시 라우터, 서비스, 저장소, 스키마 계층을 유지한다.
- 기존 구조를 깨는 대규모 재배치는 요청받지 않은 한 하지 않는다.

## 3. 코드베이스별 규칙

### Airflow / 파이프라인

- 새 DAG를 추가할 때는 기존 파일 스타일을 따른다.
- 운영 경로가 `/opt/airflow/dags/...` 기준으로 하드코딩된 부분이 많으므로 관련 경로를 바꿀 때는 호출 지점을 함께 확인한다.
- DB, S3, 외부 API 접근은 가능하면 Airflow Connection 또는 기존 서비스 래퍼 사용 방식을 유지한다.
- SQL 스키마 변경이 필요하면 `dags/sql/`과 이를 호출하는 DAG를 함께 검토한다.

### `dags/modules/`

- 이 디렉터리는 DAG에서 재사용하는 핵심 로직 계층이다. 비즈니스 로직 수정은 가능하면 여기에서 수행한다.
- Airflow 의존이 없는 모듈은 순수 파이썬 상태를 유지한다. 이미 주석에 Airflow 의존성 제거 의도가 적힌 파일들은 그 방향을 보존한다.
- 학습/분석 모듈은 입력 데이터 형식과 산출물 경로가 연쇄적으로 연결되므로 함수 시그니처 변경 시 호출부를 전부 확인한다.

### FastAPI 추천 API

- 엔트리포인트는 `app/main.py`, 라우터는 `app/api/`, 비즈니스 로직은 `app/services/`, DB 접근은 `app/repositories/`, 요청/응답 모델은 `app/schemas/`에 둔다.
- 예외는 가능한 한 서비스 계층에서 의미 있게 만들고, 라우터에서는 HTTP 변환만 담당하게 유지한다.
- 설정값은 `app/core/config.py`의 `Settings`를 우선 사용한다.

## 4. 실행과 검증

- 의존성 파일은 목적별로 분리되어 있다.
  - `requirements/requirements_airflow.txt`
  - `requirements/requirements_nlpvenv.txt`
  - `requirements/requirements_recommend_api.txt`
- 로컬 컨테이너 환경은 `docker-compose.yaml` 기준이다.
- 환경 변수는 `.env.example`을 참고한다.

변경 후 가능한 범위에서 관련 검증을 수행한다.

- 추천 API 테스트:
  - `pytest tests/test_recommend_api.py`
- FastAPI 앱 간단 확인:
  - `uvicorn app.main:app --reload`
- Airflow/Docker 환경 확인:
  - `docker compose ps`

테스트나 실행이 환경 제약으로 불가능하면, 무엇이 막혔는지 최종 응답에 명시한다.

## 5. 수정 시 주의사항

- 사용자나 다른 에이전트가 만든 변경을 임의로 되돌리지 않는다.
- `.env`, 비밀키, 실제 운영 자격 증명은 생성하거나 커밋하지 않는다.
- 외부 연동 코드 수정 시 네트워크 호출부, 재시도, 예외 처리, 타임아웃 유무를 함께 확인한다.
- 크롤링/적재 코드 수정 시 중복 적재, 날짜 범위, 인코딩, 도메인 필터 파일 사용 여부를 점검한다.
- DB 관련 수정 시 로컬 기본값과 Docker Compose 연결 정보가 서로 다른지 확인한다.

## 6. 변경 원칙

- 작은 단위로 수정하고, 영향받는 계층만 건드린다.
- 새 추상화는 반복이 실제로 보일 때만 도입한다.
- 설명이 필요한 비직관적 로직에만 짧은 주석을 단다.
- 문서가 실제 코드와 다르면 코드를 기준으로 판단하되, 필요하면 문서도 같이 갱신한다.

## 7. Git 사용 규칙

- 작업 시작 전 `git status --short`로 변경 상태를 먼저 확인한다.
- 사용자 변경이 섞여 있으면 덮어쓰지 말고, 현재 작업 범위 파일만 신중히 수정한다.
- 요청받지 않은 한 새 브랜치를 만들지 않는다.
- 커밋이나 푸시는 사용자가 명시적으로 요청한 경우에만 수행한다.
- 커밋이 필요하면 작은 단위로 나누고, 메시지는 저장소의 컨벤션을 따른다.
  - `feat:`
  - `fix:`
  - `refactor:`
  - `docs:`
  - `test:`
  - `chore:`
- 커밋 메시지 설명은 한글로 쓴다.
  - `feat: 추천 API 커서 페이지네이션 추가`
  - `fix: 뉴스 적재 중 중복 저장 버그 수정`
  - `refactor: 종목 추출 로직 분리`
  - `docs: 로컬 실행 방법 문서화`
  - `test: 추천 API 커서 검증 테스트 추가`
  - `chore: 개발 환경 설정 정리`
- `git commit --amend`, 강제 푸시, 히스토리 재작성은 사용자가 명시적으로 요청한 경우에만 한다.
- `git reset --hard`, `git checkout -- <file>`, 대량 삭제처럼 파괴적인 복구 명령은 사용하지 않는다. 꼭 필요하면 먼저 사용자 확인을 받는다.
- 최종 응답에는 필요할 때만 Git 상태 요약을 포함하고, 사용자가 보지 못한 명령 출력은 핵심만 요약해서 전달한다.
